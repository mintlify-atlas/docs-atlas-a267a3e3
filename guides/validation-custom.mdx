Beyond built-in constraints, tsgonest supports custom validation functions, global error messages, and validates complex TypeScript types including nested objects, unions, enums, and tuples.

```package-install
@tsgonest/types
```

## Custom Validation Functions

### The Validate Type

Attach a custom predicate function to any property using `Validate<typeof fn>`:

```ts title="validators/credit-card.ts"
export function isValidCard(value: string): boolean {
  // Luhn algorithm check
  let sum = 0;
  let alternate = false;
  for (let i = value.length - 1; i >= 0; i--) {
    let n = parseInt(value.charAt(i), 10);
    if (alternate) {
      n *= 2;
      if (n > 9) n -= 9;
    }
    sum += n;
    alternate = !alternate;
  }
  return sum % 10 === 0;
}
```

```ts title="payment.dto.ts"
import { Validate, Pattern } from '@tsgonest/types';
import { isValidCard } from './validators/credit-card';

interface PaymentDto {
  cardNumber: string
    & Pattern<"^[0-9]{13,19}$">
    & Validate<typeof isValidCard>;
}
```

tsgonest resolves the function's source file and emits an import statement in the generated companion:

```js title="dist/payment.dto.PaymentDto.tsgonest.js"
import { isValidCard } from './validators/credit-card.js';

export function assertPaymentDto(input) {
  // ... pattern check
  if (!isValidCard(input.cardNumber)) {
    throw new Error('cardNumber: custom validation failed');
  }
  return input;
}
```

### Custom Error Messages

Use the extended object form to provide a custom error message:

```ts
import { Validate } from '@tsgonest/types';
import { isValidCard, isValidCVV } from './validators/payment';

interface PaymentDto {
  cardNumber: string & Validate<{
    fn: typeof isValidCard;
    error: "Invalid credit card number";
  }>;

  cvv: string & Validate<{
    fn: typeof isValidCVV;
    error: "CVV must be 3 or 4 digits";
  }>;
}
```

When the validator fails, your custom message is used.

### Multiple Custom Validators

Apply multiple `Validate` constraints to a single property:

```ts
import { Validate, Format } from '@tsgonest/types';
import { isNotDisposable, isNotBlacklisted } from './validators/email';

interface SignupDto {
  email: string
    & Format<"email">
    & Validate<{
      fn: typeof isNotDisposable;
      error: "Disposable emails are not allowed";
    }>
    & Validate<{
      fn: typeof isNotBlacklisted;
      error: "This email has been blocked";
    }>;
}
```

Each validator runs independently. All must pass for the input to be valid.

### Validation Function Requirements

<Warning>
  Custom validators must be **synchronous** and **pure**:

  - ✅ Return `boolean` immediately
  - ✅ No side effects (no logging, no database calls)
  - ✅ Deterministic (same input → same output)
  - ❌ No `async` / `await`
  - ❌ No external state

  ```ts
  // ✅ Valid
  export function isValidSlug(value: string): boolean {
    return /^[a-z0-9-]+$/.test(value);
  }

  // ❌ Invalid — async not supported
  export async function isUniqueEmail(value: string): Promise<boolean> {
    const exists = await db.users.findOne({ email: value });
    return !exists;
  }
  ```
</Warning>

## Global Error Messages

The `Error<M>` type sets a global error message for a property. This message applies to **all** validation failures unless a specific constraint provides its own error.

```ts
import { Error, Format, MinLength, MaxLength } from '@tsgonest/types';

interface ContactDto {
  // Global error for all email constraints
  email: string
    & Format<"email">
    & Error<"Please enter a valid email address">;

  // Global error for all name constraints
  name: string
    & MinLength<1>
    & MaxLength<100>
    & Error<"Name is required and must be under 100 characters">;
}
```

### Error Precedence

Error messages resolve in this order:

1. **Per-constraint error** (highest priority)
2. **Global `Error<M>`**
3. **Default message** (generated by tsgonest)

```ts
import { Error, Min, Max, Int } from '@tsgonest/types';

interface AgeDto {
  age: number
    & Int
    & Min<{ value: 0; error: "Age cannot be negative" }>   // ← used for Min check
    & Max<{ value: 150; error: "Age seems unrealistic" }>   // ← used for Max check
    & Error<"Invalid age">;                                  // ← used for Int check
}
```

- `Min` fails → "Age cannot be negative"
- `Max` fails → "Age seems unrealistic"
- `Int` fails → "Invalid age" (global error)

## Complex Type Support

tsgonest validates complex TypeScript types out of the box.

### Nested Objects

Nested objects are validated recursively:

```ts
import { MinLength, Format, Pattern } from '@tsgonest/types';

interface Address {
  street: string & MinLength<1>;
  city: string & MinLength<1>;
  zip: string & Pattern<"^[0-9]{5}(-[0-9]{4})?$">;
}

interface CreateUserDto {
  name: string & MinLength<1>;
  email: string & Format<"email">;
  address: Address;           // validated recursively
  billing?: Address;          // optional — validated only if present
}
```

Generated validator:

```js
function assertAddress(input) {
  if (typeof input.street !== 'string') throw new Error('street: expected string');
  if (input.street.length < 1) throw new Error('street: length must be >= 1');
  // ... city, zip
  return input;
}

function assertCreateUserDto(input) {
  // ... name, email
  input.address = assertAddress(input.address);
  if (input.billing !== undefined) {
    input.billing = assertAddress(input.billing);
  }
  return input;
}
```

### Optional Properties

Optional properties (`?`) are validated **only if present**:

```ts
import { MinLength, Format } from '@tsgonest/types';

interface UpdateProfileDto {
  name?: string & MinLength<1>;      // validated only if provided
  bio?: string & MinLength<10>;      // validated only if provided
  website?: string & Format<"url">; // validated only if provided
}
```

If the value is `undefined`, validation passes and the property is omitted.

### Nullable Types

Nullable types accept `null` as a valid value:

```ts
import { MinLength } from '@tsgonest/types';

interface ProfileDto {
  // Accepts a string (validated) or null (accepted as-is)
  nickname: string & MinLength<1> | null;

  // Optional AND nullable
  avatar?: string | null;
}
```

When `null` is provided, constraint validation is skipped.

### Discriminated Unions

Discriminated unions are validated using the discriminant field:

```ts
import { Min, MinLength, Format } from '@tsgonest/types';

interface ClickEvent {
  type: "click";
  x: number & Min<0>;
  y: number & Min<0>;
  target: string & MinLength<1>;
}

interface NavigateEvent {
  type: "navigate";
  url: string & Format<"url">;
  referrer?: string & Format<"url">;
}

interface ErrorEvent {
  type: "error";
  code: number;
  message: string & MinLength<1>;
}

type AnalyticsEvent = ClickEvent | NavigateEvent | ErrorEvent;

interface TrackDto {
  events: AnalyticsEvent[];
}
```

Generated validator uses O(1) switch on `type` field:

```js
function assertAnalyticsEvent(input) {
  switch (input.type) {
    case "click":
      // validate x, y, target
      break;
    case "navigate":
      // validate url, referrer
      break;
    case "error":
      // validate code, message
      break;
    default:
      throw new Error('type: expected "click" | "navigate" | "error"');
  }
  return input;
}
```

<Note>
  For discriminated unions to work, every variant must share a common literal property (the discriminant). tsgonest detects common fields with literal types automatically.
</Note>

### Enum Types

TypeScript enums are validated against known values:

```ts
enum Status {
  Active = 'active',
  Inactive = 'inactive',
  Pending = 'pending',
}

interface UpdateStatusDto {
  status: Status;
}
```

Generated validator:

```js
const validStatuses = new Set(['active', 'inactive', 'pending']);
if (!validStatuses.has(input.status)) {
  throw new Error('status: expected "active" | "inactive" | "pending"');
}
```

### Tuple Types

Tuples are validated element-by-element:

```ts
import { Min, Max, MinLength } from '@tsgonest/types';

interface GeoDto {
  // [latitude, longitude]
  coordinates: [
    number & Min<-90> & Max<90>,
    number & Min<-180> & Max<180>
  ];
}

interface RangeDto {
  // [label, min, max]
  range: [
    string & MinLength<1>,
    number,
    number
  ];
}
```

Each position is validated independently with its own constraints.

### Literal Types

Literal types are validated via exact match:

```ts
interface ConfigDto {
  // Must be exactly one of these strings
  env: "production" | "staging" | "development";

  // Boolean literal
  enabled: true;

  // Numeric literal
  version: 1 | 2;
}
```

Generated validator:

```js
const validEnvs = new Set(['production', 'staging', 'development']);
if (!validEnvs.has(input.env)) {
  throw new Error('env: expected "production" | "staging" | "development"');
}
```

## Practical Example: Payment System

Here's a comprehensive example combining custom validators, error messages, and complex types:

```ts title="validators/payment.ts"
export function isValidExpiry(value: string): boolean {
  const [month, year] = value.split('/').map(Number);
  if (!month || !year) return false;
  const expiry = new Date(2000 + year, month);
  return expiry > new Date();
}

export function isValidCVV(value: string): boolean {
  return /^[0-9]{3,4}$/.test(value);
}

export function isValidIBAN(value: string): boolean {
  // Simplified IBAN validation
  return /^[A-Z]{2}[0-9]{2}[A-Z0-9]{4}[0-9]{7}([A-Z0-9]?){0,16}$/.test(value);
}
```

```ts title="payment.dto.ts"
import {
  Validate, Error, Format, Pattern, MinLength,
  Min, Max, Step, Int, MinItems, MaxItems, Trim,
} from '@tsgonest/types';
import { isValidExpiry, isValidCVV, isValidIBAN } from './validators/payment';

interface CardPayment {
  method: "card";
  cardNumber: string
    & Pattern<"^[0-9]{13,19}$">
    & Error<"Enter a valid card number">;
  expiry: string
    & Pattern<"^(0[1-9]|1[0-2])/[0-9]{2}$">
    & Validate<{ fn: typeof isValidExpiry; error: "Card has expired" }>;
  cvv: string
    & Validate<{ fn: typeof isValidCVV; error: "Invalid CVV" }>;
  name: string & Trim & MinLength<1>;
}

interface BankTransfer {
  method: "bank";
  iban: string
    & Pattern<"^[A-Z]{2}[0-9]{2}[A-Z0-9]{4}[0-9]{7}([A-Z0-9]?){0,16}$">
    & Validate<{ fn: typeof isValidIBAN; error: "Invalid IBAN" }>;
  bic?: string & Pattern<"^[A-Z]{6}[A-Z0-9]{2}([A-Z0-9]{3})?$">;
}

interface CryptoPayment {
  method: "crypto";
  wallet: string & MinLength<26> & Pattern<"^(0x)?[0-9a-fA-F]+$">;
  network: "ethereum" | "bitcoin" | "solana";
}

type PaymentMethod = CardPayment | BankTransfer | CryptoPayment;

interface LineItem {
  productId: string & Format<"uuid">;
  quantity: number & Int & Min<{ value: 1; error: "Quantity must be at least 1" }>;
  unitPrice: number & Min<0> & Step<0.01>;
}

interface CheckoutDto {
  payment: PaymentMethod;
  items: LineItem[]
    & MinItems<{ value: 1; error: "Cart is empty" }>
    & MaxItems<100>;
  couponCode?: string & Trim & Pattern<"^[A-Z0-9-]+$">;
  note?: string & MaxLength<500>;
}
```

Generated validator:

- Discriminates payment method via `switch (payment.method)`
- Validates each line item (UUID, numeric constraints)
- Runs custom validators (expiry date, CVV, IBAN)
- Applies transforms (trim)
- Uses custom error messages
- Provides structured error paths: `payment.cardNumber`, `items[0].quantity`

## Tips & Tricks

<Accordion title="Extract reusable validators">
  ```ts title="validators/common.ts"
  export function isSlug(value: string): boolean {
    return /^[a-z0-9-]+$/.test(value);
  }

  export function isFutureDate(value: string): boolean {
    return new Date(value) > new Date();
  }
  ```

  ```ts title="dto.ts"
  import { Validate } from '@tsgonest/types';
  import { isSlug, isFutureDate } from './validators/common';

  interface ArticleDto {
    slug: string & Validate<typeof isSlug>;
    publishAt: string & Validate<typeof isFutureDate>;
  }
  ```
</Accordion>

<Accordion title="Use Error<M> for user-facing messages">
  ```ts
  // ✅ User-friendly
  email: string & Format<"email"> & Error<"Please enter a valid email address">;

  // ❌ Generic error
  email: string & Format<"email">; // "expected string (email)"
  ```
</Accordion>

<Accordion title="Combine built-in and custom validators">
  ```ts
  import { Pattern, Validate } from '@tsgonest/types';
  import { isNotReserved } from './validators/username';

  username: string
    & Pattern<"^[a-z0-9-]+$">        // format check
    & Validate<typeof isNotReserved>; // business rule
  ```
</Accordion>

<Accordion title="Validate discriminated unions for polymorphic DTOs">
  ```ts
  type Notification =
    | { type: "email"; to: string & Email; subject: string }
    | { type: "sms"; to: string & Pattern<"^\\+[0-9]+$">; message: string }
    | { type: "push"; deviceId: string & Uuid; title: string };

  interface SendNotificationDto {
    notification: Notification;
  }
  ```

  O(1) discrimination via `type` field.
</Accordion>
