---
title: API Versioning Strategies
description: Global prefix and API versioning support in OpenAPI generation — URI versioning, header versioning, and per-route overrides.
---

tsgonest supports NestJS's global prefix and API versioning features. When configured, all generated OpenAPI paths reflect the correct prefixes and version segments.

## Global Prefix

Set a global prefix to prepend a path segment to all routes:

```ts title="tsgonest.config.ts"
import { defineConfig } from '@tsgonest/runtime';

export default defineConfig({
  nestjs: {
    globalPrefix: 'api',
  },
});
```

Generated paths:

| Controller Route | Generated Path |
|---|---|
| `/users` | `/api/users` |
| `/users/:id` | `/api/users/{id}` |
| `/orders` | `/api/orders` |

## URI Versioning

URI versioning adds a version segment to the URL path. This is the most common versioning strategy:

```ts title="tsgonest.config.ts"
import { defineConfig } from '@tsgonest/runtime';

export default defineConfig({
  nestjs: {
    versioning: {
      type: 'URI',
      defaultVersion: '1',
      prefix: 'v', // default — produces /v1/, /v2/, etc.
    },
  },
});
```

Generated paths:

| Controller Route | Generated Path |
|---|---|
| `/users` | `/v1/users` |
| `/users/:id` | `/v1/users/{id}` |
| `/orders` | `/v1/orders` |

### Combined with Global Prefix

When both `globalPrefix` and URI versioning are active, the global prefix comes first:

```ts title="tsgonest.config.ts"
import { defineConfig } from '@tsgonest/runtime';

export default defineConfig({
  nestjs: {
    globalPrefix: 'api',
    versioning: {
      type: 'URI',
      defaultVersion: '1',
    },
  },
});
```

Generated paths:

| Controller Route | Generated Path |
|---|---|
| `/users` | `/api/v1/users` |
| `/users/:id` | `/api/v1/users/{id}` |
| `/orders` | `/api/v1/orders` |

### Custom Prefix

Override the default `v` prefix:

```ts title="tsgonest.config.ts"
export default defineConfig({
  nestjs: {
    versioning: {
      type: 'URI',
      defaultVersion: '1',
      prefix: 'version-', // → /version-1/users
    },
  },
});
```

## Header Versioning

Header versioning passes the API version via a request header instead of the URL:

```ts title="tsgonest.config.ts"
import { defineConfig } from '@tsgonest/runtime';

export default defineConfig({
  nestjs: {
    versioning: {
      type: 'HEADER',
      defaultVersion: '1',
    },
  },
});
```

With header versioning, paths remain unchanged. Instead, each operation gets an `X-API-Version` header parameter with a const value matching the route's version:

```json
{
  "parameters": [
    {
      "name": "X-API-Version",
      "in": "header",
      "required": true,
      "schema": {
        "type": "string",
        "const": "1"
      }
    }
  ]
}
```

<Note>
  The URL paths stay clean with header versioning — `/users` remains `/users`. The version is communicated through the `X-API-Version` header on each request.
</Note>

## Per-Route Versioning with @Version

The `@Version()` decorator overrides the default version on specific methods or entire controllers:

### On a Method

```ts
import { Controller, Get, Version } from '@nestjs/common';
import type { UserDto, UserDtoV2 } from './user.dto';

@Controller('users')
export class UsersController {
  @Get()
  findAll(): UserDto[] { ... }
  // Uses default version (v1)

  @Version('2')
  @Get()
  findAllV2(): UserDtoV2[] { ... }
  // Uses v2
}
```

With URI versioning, this produces:

- `GET /v1/users` → `findAll`
- `GET /v2/users` → `findAllV2`

### On a Controller

```ts
@Version('2')
@Controller('users')
export class UsersV2Controller {
  @Get()
  findAll(): UserDtoV2[] { ... }
  // All routes use v2
}
```

All operations in this controller will be under `/v2/users`.

## Multiple Versions

You can support multiple versions simultaneously:

```ts
import { Controller, Get, Post, Version } from '@nestjs/common';
import type { UserDto, UserDtoV2, CreateUserDto } from './user.dto';

@Controller('users')
export class UsersController {
  // v1 endpoints
  @Get()
  findAll(): UserDto[] { ... }
  // GET /v1/users

  @Get(':id')
  findOne(@Param('id') id: string): UserDto { ... }
  // GET /v1/users/{id}

  // v2 endpoints
  @Version('2')
  @Get()
  findAllV2(): UserDtoV2[] { ... }
  // GET /v2/users

  @Version('2')
  @Get(':id')
  findOneV2(@Param('id') id: string): UserDtoV2 { ... }
  // GET /v2/users/{id}

  // Shared endpoint (available in both versions)
  @Post()
  create(@Body() body: CreateUserDto): UserDto { ... }
  // POST /v1/users
}
```

## Complete Real-World Example

Here's a full configuration with global prefix, URI versioning, and per-route version overrides:

```ts title="tsgonest.config.ts"
import { defineConfig } from '@tsgonest/runtime';

export default defineConfig({
  nestjs: {
    globalPrefix: 'api',
    versioning: {
      type: 'URI',
      defaultVersion: '1',
      prefix: 'v',
    },
  },
  openapi: {
    output: 'dist/openapi.json',
    title: 'My API',
    version: '2.0.0',
  },
});
```

```ts title="users.controller.ts"
import {
  Controller, Get, Post, Put, Delete,
  Body, Param, Query, Version,
} from '@nestjs/common';
import type {
  CreateUserDto, UpdateUserDto,
  UserDto, UserDtoV2,
  PaginationDto,
} from './user.dto';

@Controller('users')
export class UsersController {
  /**
   * @summary List users (v1)
   */
  @Get()
  findAll(@Query() query: PaginationDto): UserDto[] {
    return this.usersService.findAll(query);
  }
  // GET /api/v1/users

  /**
   * @summary List users (v2)
   */
  @Version('2')
  @Get()
  findAllV2(@Query() query: PaginationDto): UserDtoV2[] {
    return this.usersService.findAllV2(query);
  }
  // GET /api/v2/users

  /**
   * @summary Get user
   */
  @Get(':id')
  findOne(@Param('id') id: string): UserDto {
    return this.usersService.findOne(id);
  }
  // GET /api/v1/users/{id}

  /**
   * @summary Create user
   */
  @Post()
  create(@Body() body: CreateUserDto): UserDto {
    return this.usersService.create(body);
  }
  // POST /api/v1/users

  /**
   * @summary Update user
   */
  @Put(':id')
  update(
    @Param('id') id: string,
    @Body() body: UpdateUserDto,
  ): UserDto {
    return this.usersService.update(id, body);
  }
  // PUT /api/v1/users/{id}

  /**
   * @summary Delete user
   */
  @Delete(':id')
  remove(@Param('id') id: string): void {
    this.usersService.remove(id);
  }
  // DELETE /api/v1/users/{id}
}
```

Generated paths in the OpenAPI document:

| Method | Path | Operation |
|---|---|---|
| GET | `/api/v1/users` | `findAll` |
| GET | `/api/v2/users` | `findAllV2` |
| GET | `/api/v1/users/{id}` | `findOne` |
| POST | `/api/v1/users` | `create` |
| PUT | `/api/v1/users/{id}` | `update` |
| DELETE | `/api/v1/users/{id}` | `remove` |

## Tips & Tricks

<Accordion title="Use URI versioning for public APIs">
  URI versioning is explicit and visible in URLs:

  ```
  GET /api/v1/users
  GET /api/v2/users
  ```

  Easier for debugging and API exploration.
</Accordion>

<Accordion title="Use header versioning for internal APIs">
  Header versioning keeps URLs clean:

  ```
  GET /api/users
  X-API-Version: 2
  ```

  Better for services with many versions or gradual migration.
</Accordion>

<Accordion title="Version at the method level for gradual migration">
  ```ts
  @Controller('users')
  export class UsersController {
    @Get()           // v1 (default)
    findAll() { ... }

    @Version('2')
    @Get()           // v2
    findAllV2() { ... }
  }
  ```

  Allows both versions to coexist during migration.
</Accordion>

<Accordion title="Use semantic versioning in info.version">
  ```ts
  openapi: {
    title: 'My API',
    version: '2.1.0', // ← semantic version for the whole API
  },
  nestjs: {
    versioning: {
      defaultVersion: '2', // ← URI/header version segment
    },
  },
  ```

  `info.version` is the overall API version, while `defaultVersion` is the route version.
</Accordion>
